java agnet 学习笔记

1.应用场景

可以通过利用agent技术构建一个独立于应用程序的代理程序，类似spring的aop性质。

2.java agent和spring aop区别对比

  1. 作用级别：aop主要运用于应用程序的方法级别，而agent能作用于虚拟机级别。
  2. 组成部分：aop的实现需要目标方法和逻辑增强部分的方法，而java agent要生效需要两个工程，一个是agent代理，一个是被代理的主程序。
  3. 执行场合：aop可以运行在切面的前后或者环绕等场合，而java agent只能在preMain模式或者agentMain在主程序运行前后执行。

3.Premain模式

3.1 agent代码如下：

public class MyPreMainAgent {
    public static void premain(String agentArgs, Instrumentation inst) {
        System.out.println("premain start");
        System.out.println("args:"+agentArgs);
    }
}

3.2 pom文件配置如下：

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-jar-plugin</artifactId>
            <version>3.1.0</version>
            <configuration>
                <archive>
                    <manifest>
                        <addClasspath>true</addClasspath>
                    </manifest>
                    <manifestEntries>
                        <Premain-Class>com.cn.JavaAgentTest</Premain-Class>                            
                        <Can-Redefine-Classes>true</Can-Redefine-Classes>
                        <Can-Retransform-Classes>true</Can-Retransform-Classes>
                        <Can-Set-Native-Method-Prefix>true</Can-Set-Native-Method-Prefix>
                    </manifestEntries>
                </archive>
            </configuration>
        </plugin>
    </plugins>
</build>

参数解释说明如下：

  • Premain-Class需要包含平remain类的全类名
  • Can-Redefine-Classes：为true时表示能够重新定义class
  • Can-Retransform-Classes：为true时表示能够重新转换class，实现字节码替换
  • Can-Set-Native-Method-Prefix：为true时表示能够设置native方法的前缀

4.Agentmain模式

agentmain模式可以说是premain的升级版本，它允许代理的目标主程序的jvm先行启动，再通过attach机制连接两个jvm

4.1 agent类

需要注意！！
agent模式所使用的函数的名字是agentmain，不能随便更改，premain模式使用的函数名字是premain()

4.4 attach类

与premain模式不同，我们不能通过添加jvm参数的方式来进行连接agent和主程序，而连接需要借助com.sun.tools。attach包
下的VirtualMachine工具类，需要注意该类是sun公司自己实现，所以需要引入依赖

<dependency>
    <groupId>com.sun</groupId>
    <artifactId>tools</artifactId>
    <version>1.8</version>
    <scope>system</scope>
    <systemPath>${JAVA_HOME}\lib\tools.jar</systemPath>
</dependency>

上面的${JAVA_HOME}代表java的目录，如果找不到，可以指定本地的java的绝对路径，同时需要注意，idea的project struct下的module以及模块实际使用的是本地路径指定的那个jdk

需要统一的jdk位置示例如下：



VirtualMachine代表了一个要被附着的java虚拟机，也就是程序中需要监控的目标虚拟机，外部进程可以使用VirtualMachine的实例将agent加载到目标虚拟机中。先看一下它的静态方法attach：


public static VirtualMachine attach(String var0);

通过attach方法可以获取一个jvm的实例对象，这里传入的参数是目标虚拟机运行时的进程号pid

进程号pid可以使用jps命令查看。也就是说，我们在启动attach之前，需要启动主类，并获取其pid。

4.5 主程序

package com.cn;

import com.sun.tools.attach.AgentInitializationException;
import com.sun.tools.attach.AgentLoadException;
import com.sun.tools.attach.AttachNotSupportedException;
import com.sun.tools.attach.VirtualMachine;

import java.io.IOException;
/*
    连接agent与主类
    attach的id号用jps获得，每次运行不同
 */
public class AtachTest {
    public static void main(String[] args) {
        try {
            VirtualMachine vm = VirtualMachine.attach("30628");
            vm.loadAgent("E:\\TestJavaAgent\\target\\TestJavaAgent-1.0-SNAPSHOT-jar-with-dependencies.jar","param");
        } catch (AttachNotSupportedException e) {
            throw new RuntimeException(e);
        } catch (IOException e) {
            throw new RuntimeException(e);
        } catch (AgentLoadException e) {
            throw new RuntimeException(e);
        } catch (AgentInitializationException e) {
            throw new RuntimeException(e);
        }
    }
}

在获取到VirtualMachine实例后，就可以通过loadAgent方法可以实现注入agent代理类的操作，方法的第一个参数是代理的本地路径，第二个参数是传给代理的参数。执行AttachTest，再回到主程序AgentmainTest的控制台，可以看到执行了了agent中的代码：
